## Challenges

- Packaging smoke initially installed every internal tarball as a direct dependency, which masked whether the root `mugiew-method` package dependency chain was valid. Review caught it; smoke now installs only the root package and uses internal tarballs as overrides.
- Source manifests cannot use concrete unpublished versions during local install. The correct source strategy is `workspace:^`, relying on `pnpm pack` to rewrite ranges to semver in artifacts.
- Validation hardening had subtle process risks: unbounded output after capture, unsafe journal filename segments, and possible timestamp collisions.
- Review-project scanning had filesystem edge cases: sibling path prefix escapes, symlink escapes, binary inputs, duplicate paths, and untracked changed files.
- Discovery expectations needed product honesty. Current `/create-prd` bootstraps state; full adaptive interrogation remains deferred.
- Several review layers timed out during 7.5 and 7.3. Local verification covered the work, but review orchestration itself needs stronger bounded behavior.

## Key Learnings

1. Launch hardening is mostly negative capability.
   The highest-value work denied unsafe states, bounded execution, exposed stale approvals, and prevented overclaiming.

2. Content hashes beat file metadata for approval integrity.
   SHA-256 artifact versions are the right primitive for detecting stale approval after artifact edits.

3. Smoke tests must validate the user path, not a convenient internal path.
   Installing every internal tarball directly made the test less meaningful. Root-only install better matches `npx mugiew-method install`.

4. File scanning is security-sensitive even for local tools.
   Root containment, realpath checks, binary detection, and dedupe are required before review automation can be trusted.

5. Documentation must state implementation scope, especially for AI-native products.
   "Adaptive discovery pending" is healthier than letting future agents assume it already exists.

## Review Findings Pattern

Review worked best when it tested claims against actual behavior:

- 7.1: package smoke masked root dependency chain; fixed.
- 7.2: metadata versions missed same-size edits; fixed with content hashes.
- 7.3: output limits applied too late; fixed with Execa `maxBuffer`.
- 7.4: scan paths had containment and binary risks; fixed.
- 7.5: no actionable findings after local verification.
- 7.6: CLI output lacked direct regression coverage; fixed.

Recurring pattern: the first implementation often satisfied the happy path, while review forced the realistic autonomous path.
